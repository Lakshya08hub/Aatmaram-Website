## Page Interaction Contracts

### Pattern: List Page (Departments / Facilities / Doctors)

```
[Page header: "Departments"]  [Add Department button — accent]
[Table]
  Columns: Name | Description (truncated 60 chars) | Actions
  Actions column: [Edit icon button] [Delete icon button]
[Empty state if 0 rows]
```

**Loading state:** Render 5 Skeleton rows (matching column widths) while data fetches. No spinner.

**Empty state:** Centered in table body area.
- Heading: "No departments yet"
- Body: "Add your first department to show it on the public site."
- CTA: "Add Department" button (accent, same as page-level Add button)

**Error state (fetch failure):**
- Inline error banner below page header (not a toast)
- Copy: "Could not load departments. Check your connection and refresh."
- No empty-state UI shown when fetch errors.

**Optimistic updates:** None. All mutations are server-confirmed before UI updates. Use Sonner toast on success/failure after server round-trip.

**Delete flow:**
1. Click delete icon → AlertDialog opens immediately (no optimistic removal)
2. AlertDialog: Title "Delete [entity name]?" Body "This will remove it from the public site immediately. This cannot be undone."
3. Buttons: "Cancel" (outline) | "Delete" (destructive, red)
4. On confirm: disable both buttons, show loading spinner inside Delete button
5. On success: close dialog, remove row, show Sonner success toast
6. On failure: close dialog (row stays), show Sonner error toast

### Pattern: Add/Edit Form — Dialog (Departments, Facilities)

Triggered by: Add button (new) or Edit icon (existing — pre-fills form)

**Department form fields:**
- Name (Input, required, max 80 chars)
- Description (Textarea, required, max 500 chars)
- Image URL (Input, optional, placeholder: "https://...")

**Facilities/Services form fields:**
- Name (Input, required, max 80 chars)
- Description (Textarea, required, max 300 chars)
- Category (Select: OPD | ICU | Diagnostics | Surgery | Other)

Dialog sizing: `max-w-md`. Title: "Add Department" / "Edit Department".

**Validation:** zod + react-hook-form. Inline field errors below each input (shadcn Form pattern, already established in LoginForm). No toast for validation errors.

**Submit flow:**
1. Click "Save" → run zod validation
2. On valid: disable form, show loading spinner inside Save button
3. On server success: close dialog, refresh list, Sonner toast "Department saved"
4. On server error: re-enable form, Sonner toast "Failed to save. Try again."

**Cancel:** Closes dialog without saving. If form is dirty (user typed anything), no confirmation needed — portal CRUD forms are low-stakes.

### Pattern: Add/Edit Form — Sheet (Doctors)

Sheet side: right. Width: `sm:max-w-lg`.

**Doctor form fields:**
- Full Name (Input, required)
- Specialization (Input, required, e.g. "Paediatrics")
- Qualification (Input, required, e.g. "MBBS, MD")
- Photo URL (Input, optional)
- Bio (Textarea, optional, max 600 chars, placeholder: "Brief professional biography...")
- Availability Days (multi-select or comma-separated Input: Mon, Tue, Wed, Thu, Fri, Sat, Sun)
- Is Active toggle (Switch — controls visibility on public site)

Submit / cancel / validation / error behavior: same as Dialog pattern above.

**Loading state while opening (edit mode):** Skeleton fields while pre-fill data loads.

### Pattern: Hospital Info (Single Record Edit)

Route: `/portal/content/hospital-info`

No list view. Page renders the edit form directly (no Add button — record always exists, seeded at migration).

**Fields:**
- About Text (Textarea, large, max 1200 chars — shown on public About / Contact pages)
- OPD Timings (Input, e.g. "Mon–Sat: 9:00 AM – 1:00 PM, 5:00 PM – 8:00 PM")
- Emergency Number (Input, phone format)
- Address Line 1 (Input)
- Address Line 2 (Input, optional)
- City (Input, default "Kanpur")
- Google Maps Embed URL (Input, optional)

Submit: Full-page form with a single "Save Changes" button (accent) at bottom. No dialog/sheet.

**Submit flow:** Same server-confirmed pattern. Sonner toast on success/failure. Button shows loading spinner during save.

**Empty/error state:** If DB read fails on load, show inline error banner: "Could not load hospital info. Refresh to try again."

## Public Site Changes (No New UI)

Pages that swap from static TypeScript constants to Supabase reads:

| Page | Route | Change |
|------|-------|--------|
| Departments | `/[locale]/departments` | `getDepartments()` server call replaces static array |
| Doctors | `/[locale]/doctors` | `getDoctors()` server call replaces static array |
| Services | `/[locale]/services` | `getFacilities()` server call replaces static array |
| Contact | `/[locale]/contact` | `getHospitalInfo()` server call replaces static constants |

**Loading:** All are Server Components with `async` data fetch. No skeleton needed — Next.js Suspense boundaries handle it if added. Minimum: show existing page shell while data loads.

**Error state on public pages:** If Supabase read fails, render the page with an empty list and no error message shown to visitors (fail silently on public side — better than an error page). Log error server-side.
