# Plan 07-01: DB Migration + Public Form Wiring

## Goal

Create the Supabase table for appointment requests (with `appointment_status` enum) and wire the existing
`AppointmentForm.tsx` to a new Server Action that verifies hCaptcha server-side and inserts via service role.
After this plan, a visitor can complete the public appointment form and have their request stored in the DB.

---

## Tasks

### Wave 0 — Database Migration (blocking: must run before any code)

- [ ] APT-01: Create `appointment_status` enum and `appointment_requests` table in Supabase
  - File: `supabase/migrations/20260612_appointment_requests.sql`
  - Create this file locally as the migration record. Then paste the SQL into Supabase Dashboard → SQL Editor and run it.
  - The SQL executor must verify the table exists in Supabase before proceeding to Wave 1.

  **Migration SQL (paste verbatim into Supabase SQL Editor):**

  ```sql
  -- Phase 7: Appointment Request System
  -- Creates: appointment_status enum, appointment_requests table
  -- Includes: indexes for portal tab queries; RLS disabled (service role access only per D-02)

  -- ============================================================
  -- 1. appointment_status enum
  -- ============================================================
  CREATE TYPE appointment_status AS ENUM (
    'pending',
    'contacted',
    'confirmed',
    'cancelled'
  );

  -- ============================================================
  -- 2. appointment_requests table
  -- ============================================================
  CREATE TABLE appointment_requests (
    id               UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    patient_name     TEXT        NOT NULL,
    phone            TEXT        NOT NULL,
    preferred_doctor TEXT        NOT NULL,   -- stored as free-text name; "no-preference" allowed (D-06)
    preferred_date   DATE        NOT NULL,
    preferred_time   TEXT        NOT NULL,   -- 'morning' | 'afternoon' | 'evening' (validated by Server Action)
    reason           TEXT        NOT NULL,
    status           appointment_status NOT NULL DEFAULT 'pending',
    notes            TEXT,                  -- nullable; receptionist callback notes (D-08)
    created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
  );

  -- ============================================================
  -- 3. Indexes (primary access patterns: status tab filter, newest-first sort)
  -- ============================================================
  CREATE INDEX idx_appointment_requests_status     ON appointment_requests(status);
  CREATE INDEX idx_appointment_requests_created_at ON appointment_requests(created_at DESC);

  -- ============================================================
  -- 4. RLS — disabled; all access via service role (D-02 / D-11)
  -- ============================================================
  ALTER TABLE appointment_requests DISABLE ROW LEVEL SECURITY;
  ```

  **Verification:** After running in Supabase SQL Editor, confirm via Table Editor that `appointment_requests`
  appears with all columns and the `appointment_status` enum type is listed under Database → Types.

### Wave 1 — Server Action (depends on: APT-01 table existing)

- [ ] APT-02: Create `lib/actions/submitAppointmentAction.ts` — public Server Action with hCaptcha verify + service role insert
  - File: `lib/actions/submitAppointmentAction.ts`
  - Create the `lib/actions/` directory (new directory — no prior files exist here).
  - Mark the file `'use server'` at the top.
  - Import `createAdminClient` from `@/lib/supabase/admin` (service role client, per D-02).
  - Define and export `submitAppointmentAction` with this exact signature:

    ```
    submitAppointmentAction(input: {
      patientName: string
      phone: string
      preferredDoctor: string
      preferredDate: string
      preferredTime: string
      reason: string
      captchaToken: string
    }): Promise<{ error?: string }>
    ```

  - Implementation steps inside the action (in order):
    1. Call hCaptcha siteverify manually via `fetch('https://hcaptcha.com/siteverify', { method: 'POST', headers: { 'Content-Type': 'application/x-www-form-urlencoded' }, body: new URLSearchParams({ secret: process.env.HCAPTCHA_SECRET!, response: input.captchaToken }) })`.
       Parse response JSON as `{ success: boolean }`. If `success` is false, return `{ error: 'CAPTCHA verification failed. Please try again.' }`.
       Do NOT rely on Supabase native hCaptcha integration — that only applies to auth operations (signUp/signIn), not custom table inserts.
    2. Call `createAdminClient()` (no `await` — it returns synchronously).
    3. Insert into `appointment_requests` via `adminClient.from('appointment_requests').insert({ patient_name, phone, preferred_doctor, preferred_date, preferred_time, reason, status: 'pending' })`.
       Map camelCase input fields to snake_case column names.
       If `input.preferredDoctor === 'no-preference'`, store the string `'No Preference'` (not null — column is NOT NULL per D-06).
    4. If Supabase returns an error, throw `new Error(error.message)`.
    5. Return `{}` on success.
    6. Wrap everything in try/catch; catch returns `{ error: err instanceof Error ? err.message : 'Unknown error' }`.

  - Do NOT add any auth guard — this is a public action (D-11). No `requireAdminRole` or session check.
  - The env var is `HCAPTCHA_SECRET` (not `HCAPTCHA_SECRET_KEY`) — match the user_setup declaration above.

  **Verification:** `npx tsc --noEmit` passes with no errors on this file.
