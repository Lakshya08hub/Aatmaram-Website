<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Portal Layout (Server Component) | Authenticated-but-unauthorized or inactive users attempt access here |
| Portal Layout -> profiles (RLS) | Role/is_active read here to decide access and sidebar scope |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | Auth bypass | Portal layout guard | mitigate | getUser() check; !user -> redirect('/login'); guard runs in layout wrapping every portal route |
| T-04-04 | Elevation of Privilege | Sidebar role scoping | mitigate | Sidebar renders only ROLE_SECTIONS[role]; role read server-side from profiles, not client input |
| T-04-05 | Auth bypass | Inactive account | mitigate | !profile.is_active -> signOut() + redirect('/login') (D-05 replaces approval-queue concept) |
| T-04-08 | Spoofing | Auth guard token check | mitigate | getUser() re-validates with Auth server; getSession() forbidden in guard (acceptance gate greps for 0 getSession) |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase |

## Note on layout-level enforcement
Sidebar hiding is UX, not security. The real access control for section *content* lands when each section is built (Phases 6-10) with its own server-side role checks. Phase 4 enforces portal-entry auth (AUTH-04) and section *visibility* (AUTH-05); deeper per-section authorization is deferred to the phase that builds each section.
</threat_model>
